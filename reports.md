# Генератор заданий для контрольных работ (Moodle Coderunner)

## О проекте

Инструмент автоматической генерации уникальных вариантов заданий для контрольных работ по курсам программирования на С/С++, интегрированный в Moodle через CodeRunner.

**Технологии:** Python, C/C++, Moodle CodeRunner, Jobe  
**Репозиторий:** [moevm/mse1h2026-autogen](https://github.com/moevm/mse1h2026-autogen)

---

## Постановка задачи

Преподавателю утомительно вручную придумывать задания для контрольных работ по курсам программирования на С/С++. Нужна система автогенерации уникальных вариантов заданий через Moodle Coderunner с использованием seed для воспроизводимости.

### Категории пользователей

**Преподаватель**
- Хочет легко генерировать уникальные варианты заданий для каждого студента
- Хочет видеть, как выглядит сгенерированное задание (debug-режим)
- Хочет регулировать параметры генерации (seed, количество тестов)
- Хочет понятные сообщения об ошибках при проверке студенческих решений

**Студент**
- Получает уникальный вариант задания
- Видит понятное сообщение об ошибке (с расшифровкой, что пошло не так, какой ответ ожидался)
- Имеет ограничение по времени: 90 минут на контрольную, 5 секунд на каждую проверку

## Требования

### Направление 1 — Осенний семестр: доработка существующих заданий

Задания уже существуют в репозитории, но имеют проблемы: недостаточная вариативность, часть параметров hardcode, есть зависимость от модуля RISC-V (надо удалить).

| Задание | Тема | Что нужно доработать |
|---------|------|----------------------|
| Задание 1 | Форматный ввод/вывод (scanf, printf, hex, dec) | Добавить генерацию чисел с текстом, операции: сумма, среднее, умножение, степень |
| Задание 2 | Посимвольная обработка строк | Придумать больше вариантов операций с одним символом (например, шифр Цезаря) |
| Задание 3 | Работа с типами данных (int, float, точность) | Добавить битовые маски, возможно — комплексные числа |
| Задание 4 | Массив, разница в байтах между элементами | Исправить парсинг ответа (лишний пробел = неверный ответ), убрать программу на Си из ответа |
| Задание 5 | N строк, поиск по критерию | Варьировать критерий (макс/мин длина, количество символов и т.п.) |

### Направление 2 — Весенний семестр: новые задания

Задания предоставляет команда, заказчик согласовывает.

| Задание | Тема | Примечания |
|---------|------|------------|
| Задание 1 | Регулярные выражения | Синтаксис regex Си → Python; пример: парсинг email или номера телефона |
| Задание 2 | Списки (одно/двусвязные) | Поменять два элемента местами или обменять между двумя списками |
| Задание 3 | Шаблоны C++, переопределение, сокрытие данных | Задачи на фантазию команды |
| Задание 4 | Hashmap / словарь | Поиск коллизий, подсчёт вхождений, битовые маски |
| Задание 5 | Файловая система | Базовый: работа с файлами (существует, переименовать, бинарный I/O); Сложный: рекурсивный обход папок, поиск .txt |

### Направление 3 — Инфраструктура

- Убрать зависимость от библиотеки RISC-V (используется для генерации арифметических выражений — перевести на чистый Python)
- Реализовать debug-версию: отдельный Python-скрипт с выводом seed, тестовых данных и ожидаемых ответов; поддержка ручной установки seed

### Технический стек и окружение

- **Платформа:** Moodle Coderunner + Jobe ([github.com/trampgeek/jobe](https://github.com/trampgeek/jobe))
- **Язык генерации:** Python
- **Язык решений студентов:** C / C++
- **Формат вывода grader:** JSON (оценка + пролог/эпилог с HTML)
- **Флаги компиляции:** `-Wall -Werror -g` + sanitizers
- **Таймаут на тест:** 5 секунд
- **Лимит памяти:** до 2 ГБ
- **Количество тестов по умолчанию:** 20–30 (регулируемый параметр)

### Требования к качеству

- Все захардкоженные параметры → генерируемые через seed
- Вывод заданий использует корректные HTML-теги (таблицы с границами, перенос длинных строк)
- Ответ на задание 4 парсится с trim() пробелов
- Граничные значения должны явно покрываться в тестах
- Ошибки студента выводятся в понятном виде (не просто "wrong answer")

## Сценарии использования

Проект является внутренним набором скриптов без собственного GUI. Взаимодействие с пользователем происходит через **консольный интерфейс** (преподаватель, debug-режим) и **интерфейс Moodle** (студент).


### Консольный интерфейс (debug-режим)

```bash
# Debug-просмотр задания с конкретным seed
python generate.py --task autumn_1 --seed 42 --debug

# Генерация XML для импорта в Moodle
python generate.py --task autumn_1 --seed 42 --count 25 --export xml --out task1.xml

# Все задания семестра одним файлом
python generate.py --semester autumn --seed 42 --export xml --out autumn_all.xml

# Список доступных заданий
python generate.py --list

# Smoke-тест (проверить, что генерация не падает)
python generate.py --task spring_1 --seed 0 --validate
```

| Аргумент | По умолчанию | Описание |
|----------|-------------|----------|
| `--task` | — | Имя задания: `autumn_1`…`autumn_5`, `spring_1`…`spring_5` |
| `--semester` | — | Все задания семестра (`autumn` / `spring`) |
| `--seed` | random | Seed для воспроизводимости |
| `--count` | 25 | Количество тест-кейсов |
| `--debug` | off | Вывод условия, тестов и параметров в консоль |
| `--export` | — | Формат экспорта: `xml` |
| `--out` | `out.xml` | Путь к выходному файлу |
| `--validate` | off | Smoke-тест без вывода |
| `--list` | off | Показать список доступных заданий |

**Пример вывода `--debug`:**
```
=== DEBUG MODE ===
Seed: 42
Task: autumn_1 (форматный ввод/вывод)

--- Условие ---
Дано число N. Введите N целых чисел в формате hex.
Выведите их сумму в десятичном формате.

--- Параметры генерации ---
operation:    sum
input_format: hex
n_range:      (3, 8)
value_range:  (-255, 255)
n_tests:      25

--- Тест-кейсы (первые 3) ---
#1  stdin: "3\n0x1A 0xFF 0x2B\n"   expected: "298"
#2  stdin: "5\n0x10 0xAB ..."       expected: "..."
```

### UC-1: Преподаватель генерирует вариант контрольной

**Предусловие:** нужен XML-файл с заданиями для импорта в Moodle.

1. Запускает `python generate.py --task autumn_1 --seed 42 --count 25 --export xml`
2. Скрипт инициализирует генератор с заданным seed
3. Генерируются условие задачи, тестовые данные и ожидаемые ответы
4. Результат упаковывается в XML Moodle Coderunner и сохраняется на диск
5. Преподаватель импортирует XML в Moodle вручную

**Результат для пользователя:** готовый уникальный вариант за секунды вместо ручного придумывания. Seed обеспечивает воспроизводимость — один и тот же seed всегда даёт одинаковый вариант.

### UC-2: Преподаватель отлаживает задание (debug-режим)

**Предусловие:** внесены изменения в генератор, нужно проверить корректность без загрузки в Moodle.

1. Запускает `python generate.py --task autumn_1 --seed 42 --debug`
2. В консоль выводятся: seed, условие задачи, параметры генерации, таблица тест-кейсов
3. Визуально проверяет корректность

**Результат для пользователя:** быстрое обнаружение багов без Moodle. До этого Python-код был зашит в XML — отладка была крайне неудобной.

### UC-3: Студент решает задание в Moodle

**Предусловие:** задание импортировано в Moodle.

1. Студент видит условие задачи (HTML, сгенерированный под его seed)
2. Пишет решение на C/C++ в редакторе Moodle
3. Нажимает «Проверить» → Coderunner передаёт код и тесты в Jobe
4. Jobe компилирует: `gcc -Wall -Werror -g -fsanitize=address,undefined`
5. Программа запускается на каждом тест-кейсе (таймаут 5 сек)
6. Студент видит «Верно» или таблицу с описанием ошибок

**Результат для пользователя:** мгновенная обратная связь — студент сразу понимает, что именно не так.

### UC-4: Обработка ошибки выполнения

**Предусловие:** программа студента завершилась с ошибкой (runtime error, segfault, TLE).

1. Программа завершилась с ненулевым кодом (например, SIGSEGV = 11)
2. Grader расшифровывает сигнал через Python `signal`: `"Segmentation fault (SIGSEGV) — обращение к недопустимой памяти"`
3. Формирует JSON-ответ:
```json
{
  "fraction": 0.0,
  "prologuehtml": "<p>Тест #3 провален</p>",
  "epiloguehtml": "<table border='1'>
    <tr><th>Вход</th><td>3\n0x1A 0xFF 0x2B</td></tr>
    <tr><th>Ожидалось</th><td>298</td></tr>
    <tr><th>Получено</th><td>—</td></tr>
    <tr><th>Ошибка</th><td>Segmentation fault (SIGSEGV)</td></tr>
  </table>"
}
```
4. Студент видит таблицу в Moodle с конкретным тестом и типом ошибки

**Результат для пользователя:** вместо непонятного "wrong answer" — конкретная причина и входные данные, на которых упала программа.
